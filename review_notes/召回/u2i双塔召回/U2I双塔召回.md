# U2I 双塔召回（DSSM / Two-Tower）· 实战手记

> 视角：召回算法工程师。双塔是所有向量召回的地基。这篇不讲"什么是双塔"(那是科普)，只讲**真正决定线上效果的深水区**：负采样体系、信息瓶颈本质与改进流派、样本构造、蒸馏、向量索引工程、训练-serving 一致性。

---

## 〇、先纠正三个常见的"半对"说法

1. **"双塔用内积是因为内积可分解才能建索引"——不完整。**
   ANN 支持的是 **MIPS(最大内积检索)** 和 L2/cosine。cosine = 向量先 L2 归一化后的内积，**也能建索引**。真正不能建索引的是 **MLP matching / 任意打分函数**——因为它要 user、item 联合计算，无法拆成"每个 item 独立向量 + 一个简单距离"。所以约束不是"必须内积"，而是**打分必须能写成 `f(u)·g(v)` 这种可分解形式**。

2. **"双塔不能交叉所以表达弱"——要讲清弱在哪。**
   弱在**信息瓶颈**：user 和 item 各自压成一个向量，两者唯一的交互只发生在最后一步内积。特征交叉、user 对 item 不同字段的差异化关注，全被这个瓶颈卡死。这才是双塔一切改进的根因(见 §四)。

3. **"曝光未点击=负样本"——直接用会坏事。**
   召回的负样本主体应该是**随机/全局采样的样本**，不是曝光未点击。原因见 §三。

## 一、核心约束：为什么是这个形态

```
score(u, i) = φ_user(x_u) · ψ_item(x_i)      # 用户塔向量 · 物品塔向量
```

打分必须能拆成**两个独立向量的内积**：
- item 向量 `ψ(x_i)` 不依赖 user → 可**离线全量算好 + 建 ANN 索引**；
- 线上只算 1 个 user 向量 → MIPS 检索 → 毫秒级扛亿级物料。

**这是双塔存在的唯一理由**，也是它所有缺陷的来源。记住这条，下面所有改进都是"在不破坏可分解性的前提下，尽量缓解信息瓶颈"。

### 1.1 红线：用户塔不能用 target attention（DIN 那种）

这是双塔最容易翻车、也最能体现理解深度的点。

- **target attention(DIN)**：用户兴趣 = `attention(query=候选 item, key/value=历史行为)`，意味着**同一用户面对不同候选会算出不同的 user 向量**。
- 这直接**违反可分解性**：user 向量一旦依赖候选 item，就没法"算一个 user 向量去 ANN 检索全库"——你得对全库每个 item 重算一次 user 向量再打分，退化成 O(N) 暴力，ANN 失效。

| | 用户表示 | 可否 ANN | 适用 |
|---|---|---|---|
| **self-attention pooling**(query 来自序列自身) | 与候选无关，一个用户一个向量 | ✅ | 双塔召回 |
| **target attention**(query=候选 item) | 随候选变，一个用户 N 个向量 | ❌ | 精排 DIN(候选只几百，逐个算扛得住) |

> 结论：双塔用户塔给序列加权，**只能用 self-attention(SASRec 式)**，不能把候选 item 当 query。本质分界——召回"一次用户向量打天下(可索引)"，精排"每个候选一套用户表示(候选少不需索引)"。
> 注意 SASRec 仍是**判别式向量召回**(编码成向量再 ANN)，不是"生成式召回(GR，生成语义 ID，如 TIGER)"。

## 二、负采样：双塔的命门（这块最能看出深浅）

> 负样本是跨所有召回通路的通用核心，完整体系(easy/hard/曝光未点击/未曝光、中段分数甜区、false negative、配比、curriculum)见 [`负样本挖掘和召回效果评估.md`](../负样本挖掘和召回效果评估/负样本挖掘和召回效果评估.md)。本节是双塔视角的要点。

召回是**从全库选**，所以负样本必须近似"全库分布"，这跟精排(从曝光里学)是本质区别。负采样质量直接决定上限。

### 2.1 in-batch negatives 与它的偏置
batch 内别的样本的 item 互相当负样本，省采样、利用率高。但有致命偏置：

> **热门 item 出现在 batch 里的概率高 → 被当负样本的次数多 → 被系统性打压(popularity bias)。**

### 2.2 logQ 校正（sampled softmax correction，必考）
修正办法：打分里减掉 item 被采样的概率的 log：

```
score'(u, i) = u · v_i − log Q(i)
```

其中 `Q(i)` 是 item i 在负采样中被抽到的概率(≈在流中的出现频率)。直觉：越热门的 item `Q(i)` 越大，减得越多，抵消它"被过度当负样本"的偏置。
- **`Q(i)` 怎么估**：流式频率估计——全局计数 / 滑窗 / count-min sketch / Google 的 streaming frequency estimation(用 item 两次出现的间隔估计频率，省内存)。

### 2.3 混合负采样 MNS（Google 论文，进阶）
只用 in-batch，负样本分布 = 训练数据的 item 分布(偏热门)，**长尾覆盖不到**。
解法：**in-batch negatives + 全局均匀采样 negatives** 混合(Mixed Negative Sampling)。均匀采样补长尾，让负样本分布更接近"全库"。

### 2.4 cross-batch negatives（扩负样本量）
batch 内负样本数受限于 batch size。借鉴 MoCo，维护一个**负样本队列/memory bank**，把历史 batch 的 item 向量缓存复用，等效放大负样本规模(注意向量会因模型更新而 stale，要控队列长度)。

### 2.5 Hard Negative Mining（提质，最容易翻车）
简单负(随机)只能学个大概，**难负**(语义接近但用户没选的)才能学到细粒度边界。
- **怎么挖**：用当前模型对 query 做 ANN，取**排名靠前但非正样本**的当难负；或同类目/同价位的曝光未点击。
- **比例**：难负不能太多，常见简单负为主 + 少量难负(如难负占 10~30%)。
- **陷阱(必说)**：难负挖太狠会引入 **false negative**——那个"难负"其实用户会喜欢只是没曝光。要配合**过滤**(用户已转化的、同 session 正反馈的剔除)和**难度课程(curriculum，由易到难)**。

### 2.6 温度 τ

```
        exp(u · v⁺ / τ)
L = −log ─────────────────────────────────
         exp(u · v⁺ / τ) + Σ exp(u · v⁻ / τ)
```

（v⁺ 正样本 item 向量，v⁻ 负样本 item 向量，τ 温度）

τ 控制对比锐度：小(0.05~0.1)对比强、梯度集中在难样本、收敛快但不稳；大则欠区分。可用**可学习温度**自适应。

### 2.7 InfoNCE vs sampled softmax vs CoSENT（loss 选型，常被追问）

三者数学有重叠，出身和侧重不同：

| Loss | 出身 | 形式 | 关键点 |
|---|---|---|---|
| **sampled softmax** | 极大似然/语言模型 | 正样本 vs 采样负样本的 softmax | 目标是**对全库 softmax 的无偏估计** → **必须 logQ 校正** |
| **InfoNCE** | 对比学习 | `−log[exp(s⁺/τ)/(exp(s⁺/τ)+Σexp(s⁻/τ))]` | 自带温度 τ，常用 in-batch 负；**原始不带 logQ** |
| **CoSENT** | 文本句向量(苏剑林) | `log(1+Σexp((cos⁻−cos⁺)/τ))` | **pairwise 排序**，直接优化 cosine 的序 |

- **InfoNCE 和 sampled softmax 本质同源**：形式几乎一样(正 vs 一批负的交叉熵)。差别在"**要不要当成全库无偏估计**"——要 → 加 logQ(就成了 sampled softmax)；不要 → 纯 in-batch 对比(InfoNCE)。双塔工业主力 = in-batch softmax + logQ ≈ sampled softmax。
- **CoSENT 不同类**：不是"1 正 vs 一堆负"的 listwise softmax，而是"**正样本对相似度 > 负样本对相似度**"的 pairwise 排序，直接面向 cosine 度量。优点：比回归 cosine 绝对值(Sentence-BERT)更稳、和 spearman 评估一致；有**分级相似度标注**时(语义匹配/文本召回)特别好用。
- **选型**：大规模行为召回 → InfoNCE/sampled softmax + logQ；有人工分级相似度对 → CoSENT。

## 三、样本构造（容易被忽视，但决定天花板）

- **正样本**：按优化目标选——CTR 场景用点击，CVR/GMV 场景用转化。多目标可多种正样本加权。
- **为什么负样本不用"曝光未点击"为主**：曝光未点击是**精排已经筛过的**(都是模型认为还不错的)，拿它当召回负样本 = 让召回去学精排的残差，**和召回"从全库区分"的目标错位**，且会加剧 SSB(sample selection bias)。召回负样本主体应是随机全库样本。
- **false negative 过滤**：随机负样本里可能混入用户其实喜欢的 → 过滤用户历史正反馈过的 item。
- **样本加权**：高价值行为(购买>点击)、新 item 上采样(缓解冷启)。
- **去重/降权**：超活跃用户截断序列、刷量样本清洗。

## 四、双塔的信息瓶颈 & 改进流派（深度核心）

所有改进 = **在保持 `f(u)·g(v)` 可分解的前提下，尽量缓解瓶颈**。四条主线：

### 4.1 塔内增强（不破坏可分解性）
- **SENet / gating**：在塔内对特征做重要性加权(特征级 attention)，强化有用特征。
- **用户塔序列建模**：mean pooling → Attention/GRU/SASRec，刻画兴趣演化。
- **长短期兴趣分离**：长期兴趣(稳定) + 短期兴趣(session)分别 pooling 再融合。

### 4.2 多向量（用户侧多塔）
- **MIND / ComiRec**：用户从 1 个向量 → K 个兴趣向量，每个各自检索。双塔是 K=1 的特例。仍可分解(每个兴趣向量独立 ANN)。

### 4.3 late interaction（折中，介于双塔和 cross 之间）
- **Poly-encoder / ColBERT 式**：用户/item 各保留**多个向量**，最后做 late interaction(如 max-sim)。比单向量表达强，又比 cross 模型可索引(代价是索引和检索更复杂)。是"想要更多交叉又不想丢可索引性"的方案。

### 4.4 蒸馏：用精排/cross 模型带召回（高级，最能加分）
双塔表达弱，那就让强模型教它：
- **精排蒸召回**：用精排(cross 模型)的打分/排序当 soft label，蒸馏到双塔，把"交叉能力"以知识形式注入双塔参数。
- **对偶/自蒸馏**：拉近正样本对、增强一致性。
- 意义：召回和精排目标对齐(减少"召回觉得好、精排觉得差"的割裂)，是解决 §六坑4 的根本手段之一。

### 4.5 三塔：两种完全不同的形态（别混）

| 形态 | 第三座塔 | 解决什么 | 索引影响 |
|---|---|---|---|
| **A 冷启动三塔** | 用户塔 + **老帖塔 + 新帖塔** | 新品 ID emb 没训好、混训污染老品 | 两套 item 索引，仍可 ANN |
| **B 交叉三塔** | 用户塔 + item 塔 + **交叉塔** | 双塔信息瓶颈(补 user×item 交叉) | 交叉塔难索引，多用于粗排 |

- **A(冷启动)**：item 塔拆成老帖塔(成熟 PID emb)+ 新帖塔(只用内容/标签、**不用 PID emb**)，用户塔独立，分训后融合，线上按"是否新帖"路由。用塔隔离解决"成熟 ID 和不成熟 ID 不能混训"。详见 [`热门兜底与冷启动召回`](../热门兜底与冷启动召回/热门兜底与冷启动召回.md) §4.5。
- **B(交叉)**：加交叉塔建 user×item 交叉补瓶颈，但为保可索引常只用于粗排打分(美团/腾讯粗排三塔)。

## 五、向量索引工程（serving 深水区）

- **MIPS vs cosine**：训练用啥度量，检索就用啥；cosine 必须**两侧都 L2 归一化**再入库/检索。
- **HNSW**：图索引，召回率高、延迟低，内存大。调 `M`(图连接度)、`efConstruction`(建图质量)、`efSearch`(检索精度/延迟权衡)。
- **IVF-PQ**：倒排 + 乘积量化，省内存、适合超大库，代价是量化误差(召回率略降)。`nlist`/`nprobe`/PQ 段数权衡。
- **量化的精度损失**：PQ 会让向量失真，**训练度量和量化误差要一起考量**，必要时用 OPQ/重排(粗召回多取一些再用精确内积重排)。
- **多路融合**：多个召回通路分数量纲不一致，merge 前要**归一化/分位归一**，否则双塔分数和 I2I 分数没法比。

## 六、常见坑（实战）

**坑 1 · 热门霸屏** → logQ 校正 + 混合负采样补长尾 + 召回后热度打散。

**坑 2 · 长尾召回不出来** → 只用 in-batch(负样本偏热门)导致；上 MNS 全局均匀负采样。

**坑 3 · 训练-serving 不一致** → ①度量不一致(训练内积/线上 cos)②归一化漏了③特征穿越(用了线上拿不到的未来特征)④负样本分布和 serving 全库分布差太远。逐项对齐。

**坑 4 · 离线 Recall 高、线上不涨** → 召回和精排割裂(召回觉得好的精排不认)。排查进精排占比、与精排打分相关性；根治靠**精排蒸召回**对齐目标。

**坑 5 · 向量坍缩(全趋同)** → 负样本太弱/τ 不当/batch 太小。加难负、调 τ、扩 batch、cross-batch negatives。

**坑 6 · 两塔版本错位** → user 向量(实时)和 item 向量(T+1)用了不同模型版本，空间对不上 → 同版本同步更新，或冻结 item 塔只更用户塔。

**坑 7 · item 塔频次偏置** → 高频 item 训练充分、长尾欠拟合。item id 频次正则 / 长尾上采样 / side-info 泛化。

## 七、常见问答（带追问，深版）

**Q1：双塔为什么不能做特征交叉？只能用内积吗？**
约束不是"必须内积"，是**打分必须可分解成 `f(u)·g(v)`**(内积/归一化后 cos 都行)，这样 item 向量才能离线建索引。MLP matching 要联合计算所以不行。代价是信息瓶颈——user/item 只在最后一步交互。
> 追问·那怎么缓解瓶颈又不丢可索引性？ → 塔内增强(SENet/序列)、多向量(MIND)、late interaction(Poly-encoder)、精排蒸馏。

**Q2：in-batch 负采样的偏置是什么，怎么修？**
热门 item 在 batch 里出现多→被过度当负样本→被打压。用 **logQ 校正**(打分减 logQ(i))。
> 追问·只 in-batch 够吗？ → 不够，负样本分布偏热门、长尾覆盖差，要加**全局均匀采样**(MNS)；想扩负样本量用 **cross-batch negatives**(MoCo 式队列)。

**Q3：召回的负样本能用曝光未点击吗？**
不应作主体。曝光未点击是精排筛过的，拿来当召回负样本会让召回学精排残差、和"从全库区分"目标错位、加剧 SSB。召回负样本主体用随机全库样本，难负可少量用同类目曝光未点击。

**Q4：Hard Negative 怎么挖？有什么风险？**
用当前模型 ANN 取"排名高但非正"的当难负，或同类/同价曝光未点击。风险是 **false negative**(其实用户喜欢只是没曝光)，要过滤已正反馈样本 + 控难负比例 + curriculum 由易到难。

**Q5：离线指标好线上不涨，怎么排查？**
进精排占比 → 与精排打分相关性 → 多路 merge 量纲 → 是否被重排去重 → 训练/serving 一致性。根因常是**召回精排割裂**，用精排蒸召回对齐。

**Q6：双塔和 MIND / late-interaction 关系？**
双塔=单向量(K=1)；MIND=用户侧多向量(多兴趣)；Poly-encoder/ColBERT=多向量 + late interaction。都是在缓解信息瓶颈，复杂度和可索引性依次权衡。

**Q7：为什么需要温度 τ？**
缩放 logits 控制对比锐度，影响梯度在难/易样本上的分配。小 τ 聚焦难样本、收敛快但不稳。可学习温度更省心。

**Q8：双塔用户塔能不能用 DIN 的 target attention？**
不能。target attention 让 user 向量依赖候选 item，破坏可分解性、无法 ANN(退化成全库逐个打分)。只能用 self-attention(SASRec 式)给序列加权。这是召回(一次用户向量打天下)和精排(每候选一套表示)的本质分界。详见 §1.1。

**Q9：InfoNCE、sampled softmax、CoSENT 区别？召回选哪个？**
InfoNCE/sampled softmax 同源(正 vs 一批负的 softmax)，差在要不要 logQ 无偏校正；CoSENT 是 pairwise 排序、直接优化 cosine 序。行为召回主力 InfoNCE/sampled softmax+logQ，有分级相似度标注用 CoSENT。详见 §2.7。

**Q10：三塔是什么？**
两种：冷启动三塔(老帖塔+新帖塔，隔离不成熟 ID)；交叉三塔(加交叉塔补瓶颈，多用于粗排)。详见 §4.5。

## 八、选型一句话

- 要快、稳、好部署、做基建 → 双塔首选。
- 兴趣发散要多样性 → 用户侧上 MIND/ComiRec。
- 想更强交叉又要可索引 → late interaction(Poly-encoder)。
- 召回精排割裂严重 → 精排蒸召回。
- 要真·细粒度 user×item 交叉 → 那是精排，召回别硬上。

## 九、监控指标

- 离线：Recall@K、HitRate@K、与精排打分的相关性(召回质量代理)。
- 在线：进精排占比、长尾占比、热门集中度(基尼)、与其他通路重叠度、pvctr/uvctr。
- 健康度：向量分布(避免坍缩)、item 覆盖率、两塔版本一致性。
