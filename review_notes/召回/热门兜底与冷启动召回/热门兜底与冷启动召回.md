# 热门兜底 & 冷启动召回 · 实战手记

> 视角：召回算法工程师。这条通路不"高级"，但**保下限、保新用户体验、保新品起量**，是线上稳定性的安全网。面试容易被问"新用户/新品怎么办"。

---

## 一、一句话讲清

- **热门兜底**：当个性化通路召回为空/不足/置信度低时，用热门/高质量 item 补齐，保证任何请求都有结果。
- **冷启动召回**：给**新用户**(无行为)、**新 item**(无曝光)专门设计的召回，解决"个性化模型没数据可用"的问题。

---

## 二、为什么必须有它

- 个性化通路对**新用户/新品/低活用户**几乎失效(没行为、没共现、没向量)；
- 系统故障/超时降级时要有兜底，不能给空结果；
- 新品需要"探索曝光"才能积累数据，否则永远冷启(马太效应)。

---

## 三、热门兜底怎么做（别只会 count）

- **不是简单按点击量排序**——纯点击量会被刷量/标题党污染，且强者恒强。
- **质量分热门**：用 CTR/CVR/完播 + 平滑(Wilson 置信下界 / 贝叶斯平滑)，避免小样本高估。
- **分群热门**：按 性别/年龄/地域/类目 分桶出榜，比全局热门更相关。
- **时间窗口**：实时热门(抓趋势) + 长期热门(保稳定)结合，加时间衰减。
- **去偏**：位置偏置、曝光量纲归一，防"曝光多→点击多→更曝光"的正反馈。

---

## 四、冷启动怎么做

### 4.1 新用户冷启
- **人群/属性兜底**：按注册信息(年龄/性别/地域/机型)给同群体热门。
- **快速试探**：前几刷用多样化探索，快速收集反馈(bandit/Thompson sampling)。
- **端上实时**：用当次 session 行为即时建模(实时序列召回)。

### 4.2 新 item 冷启
- **内容/多模态 embedding**：靠图文内容召回，不依赖行为(见多模态召回笔记)。
- **side info graph(EGES)**：靠类目/品牌属性给新品合理向量。
- **强制探索流量(E&E)**：给新品保量曝光，bandit 控制探索成本。

### 4.3 物品表示"三件套"——冷启动到底靠谁（深水区）

一个 item 的向量表示由三部分拼成，**来源/成熟时机/存取方式完全不同**，搞清楚这个才懂冷启动靠什么：

| 表示组成 | 怎么来 | 成熟靠什么 | 冷启可用 |
|---|---|---|---|
| **物品 ID embedding** | 模型自己的 embedding table，随训练更新 | 该 item 自身行为累积 | ❌ 新品行为少没训好 |
| **side info embedding**(类目/作者/标签) | 也是模型 embedding table，但**和全体老品共享** | **共享训练**(海量老品一起训) | ✅ 成熟可用 |
| **内容/多模态 embedding** | **离线预训练编码器**抽图文视频 | 预训练模型，一入库就有 | ✅ 即时可用 |

> 冷启动本质：**新品 ID embedding 废了，只能靠"共享的 side info + 预训练内容向量"泛化撑着。**

**多模态 embedding 的计算与存取链路（高频追问）**

- **算**：发布即触发(Kafka → 内容抽取服务)，图用 ViT/CLIP-image、文用 BERT/CLIP-text、视频抽帧+时序 pooling，产出定长向量(如 512 维)。编码器一般**冻结**、周期性离线刷新；少数端到端微调(贵)。
- **存**：① 在线 KV(Redis/自研，`item_id→emb`，float16 压缩)给线上取；② 向量库(Faiss)直接建**内容 I2I 索引**；③ 离线特征仓(Hive，按天快照)给训练 join。
- **取**：训练时按 `item_id` join 出 emb 列(取**当时快照**防穿越)；线上两种用法——(a) 当 item 塔输入特征，过投影 MLP 适配到推荐 emb 维度(512→64)；(b) 直接当召回向量灌 Faiss 做新品内容召回。
- ⚠️ **side info 不是预算的向量**，是模型 embedding table 共享查表(lookup)；它"成熟"来自共享，不是预计算，别和多模态向量搞混。

### 4.4 hash id / 动态 embedding——海量+动态 ID 的工程编码

解决的是另一个问题：**海量(亿级) + 动态新增的 ID 怎么塞进固定大小 embedding table。**

- **普通 embedding table 的痛点**：要维护 `id→连续index` 词典，静态，**新 id 没槽位**；亿级全给独立槽内存爆；长尾 id 学不好。
- **hashing trick**：`slot = hash(id) % B`，免词典、表固定、**新 id 即时有槽**。代价是**hash 冲突**(不同 id 撞同槽、语义污染)。
- **降冲突的工业做法**：
  - **QR embedding(Quotient-Remainder，DLRM)**：用"商"和"余数"两个 hash 查两张表再组合，近似还原唯一性，省内存又降冲突；
  - **multi-hash / 双哈希**：多 hash 映射多槽，emb 求和(Bloom 思想)；
  - **频次感知**：高频 id 给专属槽，长尾走 hash 桶；
  - **dynamic embedding(弹性 embedding)**：哈希表动态扩容真正给每 id 槽位，配**准入**(频次门槛)和**淘汰**(LRU/过期)防 OOM，不靠固定桶。
- **和冷启动的关系（关键）**：

> **hash 解决"新 id 有没有位置进模型"；冷启动泛化解决"没行为时这个位置用什么信息撑着"。两者正交互补。**

新品 hash 到一个槽 ≠ 训好了(随机初始化或冲突共享)。所以行为来之前靠**内容向量 + 类目/作者 side info** 泛化，行为攒够后 hash 槽的 ID emb 才训出个性化。

### 4.5 三塔冷启动架构（用户塔 + 老帖塔 + 新帖塔，工业案例）

**动机**：原双塔里新帖(如 24h 内 UV<100)的 **PID(帖子 ID)Embedding 没训好**，混进同一个 item 塔会①新帖被错误打压/乱曝光②**污染老帖训练**。

**方案**：item 塔拆两座——
- **老帖塔**：成熟 PID Embedding + 行为特征，保原效果；
- **新帖塔**：**不用 PID Embedding**，只用内容特征 + 标签特征(SENet + Wide&Deep)，靠内容泛化；
- **用户塔**独立，三塔分别训练后融合；线上用 Redis 实时拿"是否新帖"标签做**路由**。

**为什么有效**：用"塔的隔离"解决"成熟 ID 和不成熟 ID 不能混训"；新帖走内容泛化、老帖不受污染。典型收益：人均停留时长 +1%量级，新帖 24h 起量(破百 uv 数量)+5%量级。

> 对比另一种"三塔"(用户塔 + item 塔 + **交叉塔**)：那是为补**信息瓶颈**做 user×item 交叉(多用于粗排，交叉塔难索引)，和这里的冷启动三塔目的不同，别混。

### 4.3 探索与利用（E&E，高频考点）
- **ε-greedy**：以 ε 概率随机探索。简单粗暴。
- **UCB**：给不确定性高(曝光少)的 item 加探索加成 `score + 置信区间宽度`。
- **Thompson Sampling**：按后验分布采样，平衡探索利用，工业常用。

## 五、版本迭代

### v0 · 全局点击量热门
问题：刷量污染、强者恒强、新用户不相关。

### v1 · 质量分 + 平滑 + 分群
CTR/CVR + Wilson 平滑 + 分人群榜。效果：相关性、抗噪提升。

### v2 · 冷启专项
新用户走人群兜底 + 实时 session；新品走多模态/EGES + 探索流量。效果：新用户留存、新品起量改善。

### v3 · E&E 系统化
Thompson Sampling/UCB 控制探索，量化探索成本与长期收益。

## 六、常见坑

**坑 1 · 纯点击量被刷/标题党污染**
解：质量分(CTR/CVR/完播) + 平滑 + 异常检测过滤刷量。

**坑 2 · 小样本高估(新品偶然高 CTR 就霸榜)**
解：Wilson 置信下界 / 贝叶斯平滑，按置信下界排序。

**坑 3 · 强者恒强，新品永远冷启**
解：强制探索流量 + E&E，给低曝光 item 探索加成。

**坑 4 · 兜底占比过高(个性化被稀释)**
现象：兜底召回挤占个性化坑位。解：兜底只在不足时补齐、限配额、降权排序。

**坑 5 · 探索成本失控(乱推伤体验)**
解：探索流量限额、只在低风险位探索、Thompson 控制比例。

## 七、常见问答（带追问）

**Q1：新用户冷启怎么做？**
人群属性兜底(同群体热门) + 当次 session 实时行为建模 + 多样化探索快速收敛兴趣。

**Q2：新 item 冷启怎么做？**
内容/多模态 embedding 召回(不依赖行为) + EGES side info 给向量 + 强制探索流量积累数据。
> 追问·探索成本怎么控？ → bandit(Thompson/UCB)量化不确定性，限制探索流量比例，只在长尾位探索。

**Q3：热门榜怎么排，能只用点击量吗？**
不能。点击量被刷量/标题党污染且强者恒强。要用质量分(CTR/CVR)+小样本平滑(Wilson/贝叶斯)+分群+时间窗。

**Q4：ε-greedy / UCB / Thompson 区别？**
ε-greedy 固定概率随机探索(简单但盲目)；UCB 给不确定性高的加确定性探索加成；Thompson 按后验采样(随机但有依据)。工业常用 Thompson。

**Q5：兜底召回会不会拉低整体效果？**
会，如果占比失控。要点：仅在个性化不足时补齐、限配额、排序降权，并监控兜底占比。

**Q6：冷启动用的内容 embedding 是提前算好的吗？怎么取？**
是。发布即触发(Kafka→抽取服务)用预训练编码器离线抽好，存 Redis(`item_id→emb`)/Faiss/离线特征仓。训练 join、线上 KV 查或直接 Faiss 召回。side info(类目/作者)不是预算向量，是模型 embedding table 共享 lookup，靠共享训练成熟。
> 追问·内容向量维度和推荐 emb 不一样怎么办？ → 过投影 MLP 适配(512→64)。

**Q7：hash id 是什么，和冷启动什么关系？**
hash 把海量/动态 ID 映射到固定桶(`hash(id)%B`)，免词典、新 id 即时有槽，代价是冲突(用 QR/multi-hash/动态 embedding 缓解)。但 **hash 给"槽位"不等于训好了**——冷启动期仍靠内容/side info 泛化，行为攒够后槽位 ID emb 才个性化。两者正交互补。

**Q8：为什么要搞三塔(老帖塔+新帖塔)，不直接一个 item 塔？**
新帖 PID emb 没训好，混进同一塔会被错误打压且污染老帖。拆新帖塔(只用内容/标签、不用 PID emb)做隔离，新帖走内容泛化、老帖保原效果。

---

## 八、选型一句话

- 任何线上系统都**必须有**热门兜底(保下限/降级)。
- 新用户多 → 人群兜底 + 实时 session + 探索。
- 新品多 → 多模态/EGES + E&E 探索流量。

---

## 九、监控指标

- 兜底占比(过高预警)、新用户次留/转化、新品起量速度与曝光基尼系数、探索流量 ROI、整体 pvctr/uvctr。
