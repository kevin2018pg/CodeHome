# 序列召回（SASRec / GRU4Rec / BERT4Rec）· 实战手记

> 视角：召回算法工程师。序列召回把用户行为当**有序序列**建模，直接预测"下一个 item"，对**时效性/趋势/即时意图**最敏感。是双塔用户塔的进阶形态。

---

## 一、一句话讲清

把用户行为序列 `[i1, i2, …, it]` 输入序列模型（RNN/Transformer），输出一个"下一个 item"的用户向量，去 item 库 ANN 检索 TopN。**强调顺序和近因**——刚看的比很久前看的重要。

> 和双塔 mean pooling 的区别：pooling 丢了顺序，序列模型保留顺序和演化，能抓住"用户正在干什么"。

---

## 二、三个代表模型

- **GRU4Rec**：RNN/GRU 编码序列，最后隐状态当用户向量。简单、时序强，但长序列遗忘、不能并行。
- **SASRec（主力）**：单向 Self-Attention（causal mask，只看左边），用最后位置输出预测下一个。并行、长序列友好、效果好，工业最常用。
- **BERT4Rec**：双向 Attention + 完形填空(Mask)训练。表征更强，但双向和"预测下一个"的召回目标不完全契合，线上用得比 SASRec 少。

---

## 三、训练要点

- **next-item 预测**：用位置 t 的输出预测 t+1 的 item，sampled softmax / in-batch 负采样。
- **causal mask（SASRec）**：每个位置只能看到它之前的行为，防止信息泄漏。
- **位置编码**：可学习位置 embedding；也有用时间间隔编码（TiSASRec）建模行为间隔。
- **序列特征**：除了 item_id，还可拼类目/行为类型/时间间隔。
- **负采样**：未曝光随机负打底 + 中段难负提质 + logQ 校正，体系见 [`负样本挖掘和召回效果评估.md`](../负样本挖掘/负样本挖掘和召回效果评估.md)。

**序列长度怎么选（调研结论）**

先说结论：**不存在"越长越好"，是边际递减 + 算力/时延约束下的权衡**。工业上几乎都拆成"短期序列 + 长期序列"两套来处理。

| 序列类型 | 常用长度 | 说明 |
|---|---|---|
| 短期/即时意图 | **最近 20~50**（多数）、上限 100~200 | SASRec 原论文 MovieLens 用 maxlen=200、Beauty 用 50；工业短视频/电商短期通路常取最近 50~100。这一段是 attention 全连接建模，长度受 O(L²) 时延限制。 |
| 长期兴趣 | 几百 ~ 上千、极端到万级 | 阿里 SIM/ETA、MIMN 这类，长期序列不做全 attention，而是**先检索后建模**。 |

要点：

- **边际递减**：50→100→200 有提升但越来越小，超过几百后全 attention 收益很低、时延却线性/平方涨。所以"再长"靠检索而不是堆 attention。
- **超长序列两阶段（SIM/ETA）**：GSU（General Search Unit）先用 target item 的类目/向量从万级历史里硬搜（同类目）或软搜（向量 top-k）出 50~100 条相关行为，ESU（Exact Search Unit）再对这 50~100 条做 attention 建模。等于"长期历史"压成"和当前 target 相关的短序列"。
- **默认配置（可直接背）**：短期取最近 **50~100**；长期走 SIM，检索 top **50~100** 再建模。先用 50 起步，看 Recall@K/时延再加长。
- **冷启/短序列用户**：行为不足 N 时 padding + mask，别让 padding 污染；太短（<3~5）直接退化到双塔/热门兜底。

## 四、版本迭代

### v0 · GRU4Rec
GRU 编码，最后隐状态召回。问题：长序列遗忘、训练慢。

### v1 · SASRec
换 Self-Attention + causal mask。效果：长序列、并行、召回质量提升，成为主力。

### v2 · 时间感知
加行为时间间隔编码（TiSASRec 思路），区分"连续浏览"和"隔天回访"。效果：时效性更准。

### v3 · 多目标/多行为序列
区分点击/加购/购买多行为序列，或多兴趣化(和 MIND 结合，输出多个向量)。效果：意图刻画更细。

## 五、常见坑

**坑 1 · 长序列遗忘 / 截断丢信息**
解：SASRec 替代 RNN；长序列分段或取最近 N + 长期兴趣单独建模(SIM/ETA 思路)。

**坑 2 · 近因偏置过强，只推刚看的**
现象：一直推同一类。解：长短期兴趣融合；和其他通路(I2I/双塔)多样性互补。

**坑 3 · 冷启/短序列用户效果差**
解：短序列走双塔/热门兜底；行为不足时退化。

**坑 4 · 训练-serving 序列不一致**
原因：离线用完整序列、线上用实时截断序列，长度/顺序对不上。解：对齐序列构造逻辑和最大长度。

**坑 5 · 位置/时间泄漏**
原因：mask 写错看到未来。解：严格 causal mask，单测验证。

## 六、常见问答（带追问）

**Q1：序列召回比双塔 mean pooling 好在哪？**
保留行为顺序和近因，能抓即时意图和兴趣演化；mean pooling 把序列拍平丢了顺序。

**Q2：SASRec 和 BERT4Rec 区别，召回选哪个？**
SASRec 单向(causal)、目标就是"预测下一个"，和召回天然契合；BERT4Rec 双向 + 完形填空，表征强但和 next-item 召回目标不完全一致。**召回主力是 SASRec**。
> 追问·为什么 BERT4Rec 双向反而不一定好？ → 召回要预测"未来下一个"，双向训练时看到了后文，和线上"只有历史"的场景有 gap。

**Q3：长序列怎么处理？**
取最近 N(短期) + 长期兴趣单独检索(SIM 两阶段：先用 target/类目从长期序列检索相关行为，再建模)。SASRec 本身比 RNN 更扛长序列。

**Q4：序列召回和 MIND 关系？**
都建模行为序列。MIND 输出多个兴趣向量(多兴趣)，序列召回通常输出单个"下一个"向量(单兴趣偏即时意图)。可结合：序列编码 + 多兴趣输出。

**Q5：怎么防信息泄漏？**
SASRec 严格 causal mask(只看左侧)；训练/线上序列构造一致；时间特征不能用未来。

## 七、选型一句话

- 时效性强、即时意图重(短视频/电商) → 序列召回(SASRec)。
- 兴趣发散要多样性 → MIND/多兴趣。
- 要稳要快要基建 → 双塔。
- 实践常并行：序列抓即时 + 双塔抓整体 + I2I 抓相关。
