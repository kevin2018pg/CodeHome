RankMixer 算法工程师深度手册

这份材料面向已经做过 MMoE / PLE / CTR-CVR 多任务建模的算法工程师。 它不把 RankMixer 当“新名字”背，而是从工业排序的计算约束、特征交互、模型扩容、离线训练和线上 serving 来理解。

论文：RankMixer: Scaling Up Ranking Models in Industrial Recommenders。 论文摘要强调两个痛点：工业推荐的训练/推理成本受严格延迟和高 QPS 限制；很多手工特征交叉模块继承自 CPU 时代，GPU MFU 低、扩展性差。RankMixer 用 multi-head token mixing 替代 quadratic self-attention，并用 per-token FFN 保留不同特征子空间建模，同时扩展到 Sparse-MoE。

为什么需要 RankMixer 数学抽象 Block 结构 和 MMoE/PLE 对比 Sparse-MoE 离线训练 线上推理 面试题 速记

1. 为什么工业排序需要 RankMixer

特征交互越来越重

推荐精排不只是 user/item 两塔，而是用户画像、候选物料、上下文、序列、交叉统计、上游分数等多域特征共同建模。

问题 手工交叉模块很多，但结构碎，GPU 利用率低。

Transformer 直接上会贵

Self-attention 对 token 数是二次复杂度，并且 QK/softmax 对线上 P99 不友好。

取舍 RankMixer 用静态 token mixing 换吞吐和稳定延迟。

模型扩容要看 ROI

工业排序不是参数越大越好，而是单位延迟、单位 GPU 成本带来的线上收益。

目标 让参数规模、MFU、线上延迟之间更划算。

你可以把 RankMixer 的出发点记成：不是为了“更花哨地做多任务”，而是为了把推荐排序里的多域特征交互模块统一成一个 GPU 友好、可堆叠、可扩容的 backbone。

2. 数学抽象：从特征到 token

假设一个曝光样本包含多个特征域：

x = {x_user, x_item, x_context, x_sequence, x_cross}

每个域先通过各自的 tokenizer 得到一个 token：

e_i = Tokenizer_i(x_i), i = 1..T
E = [e_1; e_2; ...; e_T] ∈ R^{T × d}

在我们当前仓库里，T=5，对应：

Token 字段域 处理方式 建模含义
user_profileunionid、会员、设备、历史统计ID/枚举 hash embedding，数值 projection用户长期偏好和活跃状态
candidatecontent_id、author_id、price、质量分ID embedding + 数值 projection候选物料自身表达
context流量来源、位置、时间、会话深度枚举 embedding + 数值 projection本次请求场景
sequence点击率、观看深度、跳过率等聚合数值 projection短期行为偏好
cross用户-物料相似、价格 gap、意图匹配数值 projection显式交叉统计

RankMixer 的核心不是把所有特征拼成一个长向量再 MLP，而是保留 token 维度，让模型知道“这块表示来自哪个特征域”。

3. RankMixer Block：Token Mixing + Per-token FFN

Token Mixing 到底学什么

它不是 attention，不会根据样本动态生成 token-token 权重。它通过规则化的维度重排，把不同 token 的 channel 混到一起，让后续 FFN 看到跨域信息。

优点是计算稳定、并行友好、延迟可控；代价是动态交互能力弱于 attention。

Per-token FFN 为什么关键

如果 token mixing 只做重排，模型还需要非线性变换来吸收交互后的信息。Per-token FFN 允许每个特征域拥有自己的变换空间。

这对应推荐里的异构特征：ID、上下文、序列统计和交叉统计不应该完全共用一个 FFN。

H = Mix(E) + E
Z_i = FFN_i(LN(H_i)) + H_i
y_task = Head_task(Pool(Z))

5. Sparse-MoE 和 DTSI

RankMixer 进一步扩到 Sparse-MoE，核心目标是提高参数容量，但不让单次推理成本线性增长。

FFN_i(h_i) = Σ_j gate_{i,j}(h_i) · Expert_{i,j}(h_i)

Dense Training

训练时让更多专家参与，更多专家获得梯度，缓解“专家训练不充分”。

收益 专家更充分训练，路由更稳定。

Sparse Inference

推理时只取 top-k experts，减少单次请求计算量。

风险 top-k 路由倾斜、跨设备通信、P99 抖动。

算法工程师要能说出的 MoE 监控

  • expert activation ratio：每个专家被选中的比例。
  • routing entropy：路由是否过度集中。
  • load balance loss：专家负载是否均衡。
  • per-expert gradient norm：是否有专家学不到。
  • 线上 top-k 稳定性：同类请求是否路由过度抖动。
  • P99 latency：是否因专家分布或跨设备通信产生长尾。

面试陷阱：不要把 RankMixer 的 Sparse-MoE 和 MMoE 的 task gate 混为一谈。MMoE gate 是任务视角，RankMixer MoE 是 backbone 扩容和 per-token FFN 替换视角。

6. 离线训练：样本、标签、损失、评估

6.1 样本组织

工业精排通常是 request 内多个候选：

D = {(request_id, unionid, item_id, features, ctr_label, cvr_label)}

训练可以 pointwise，但验证必须 request-wise。因为线上展示是 list 排序，不是孤立样本分类。

6.2 CTR/CVR 多任务目标

L = BCE(logit_ctr, y_ctr) + α · BCE(logit_cvr, y_cvr) + β · L_aux_moe

如果 CVR 极稀疏，可以考虑：

  • 提高 CVR loss 权重 α。
  • 按转化样本重采样或加权。
  • 引入 ESMM 思路：pCTCVR = pCTR · pCVR。
  • 处理 delayed feedback，避免未回传转化被误当负样本。

6.3 评估指标

指标看什么注意点
CTR AUC点击区分能力可能和最终 GMV/转化目标不一致
CVR AUC转化区分能力样本稀疏，波动大
GAUC按用户/request 加权排序能力比全局 AUC 更贴排序
NDCG@K列表前 K 排序质量和线上展示更接近
Calibration预估概率是否可信影响融合公式和出价/价值预估

7. 线上推理：算法工程师必须懂的工程约束

线上打分公式

常见不是直接用一个 sigmoid，而是业务目标融合：

score = pCTR
score = pCTR * pCVR
score = pCTR * pCVR * price
score = w1*pCTR + w2*pCVR + w3*quality - w4*risk

训推一致 Checklist

  • hash bucket 数和 hash 函数一致。
  • 数值归一化参数一致。
  • 缺失值默认值一致。
  • 枚举值大小写和映射一致。
  • 序列截断窗口一致。
  • 模型版本和 feature schema 版本绑定。

8. 面试题：算法工程师版

Q1：RankMixer 相比 DCN/DeepFM 的核心优势是什么？

DCN/DeepFM 更偏显式或半显式特征交叉，适合中等规模特征交互；RankMixer 把多域特征组织成 token，通过可堆叠 block 做统一交互，更适合工业大模型扩容和 GPU 并行。

Q2：为什么 RankMixer 不直接用 attention？

工业精排要在高 QPS 和严格 P99 下服务。Attention 的 token-token 动态权重计算和 softmax 会带来更高成本。RankMixer 用静态 token mixing 保留跨域信息流动，同时减少复杂度和延迟波动。

Q3：Token Mixing 没有参数，会不会表达力不足？

它本身确实弱于 attention 的动态交互，但后面的 per-token FFN 提供非线性变换。RankMixer 的设计不是追求单模块最强，而是在表达力、吞吐、MFU 和线上延迟之间取工业最优解。

Q4：RankMixer 如何处理多任务？

RankMixer backbone 输出共享表示，后面可以接 CTR/CVR 多个 head，也可以接 MMoE/PLE。RankMixer 解决的是 backbone 的特征交互，MMoE/PLE 解决的是任务共享和分离。

Q5：CVR 很稀疏时怎么训？

可以提高 CVR loss 权重、做转化样本重采样、引入 ESMM 的 pCTR/pCVR 链式建模、处理 delayed feedback，并重点看 CVR 分桶校准和 NDCG@K，而不只看 AUC。

Q6：Sparse-MoE 上线最大风险是什么？

不是离线指标，而是路由和系统稳定性：专家负载倾斜、跨设备通信、P99 抖动、部分专家训练不足、fallback 复杂。上线必须配 expert 激活监控和延迟监控。

Q7：离线提升为什么线上可能不涨？

可能因为候选集不一致、线上特征新鲜度不同、分数校准差、业务融合公式不匹配、延迟导致降级、特征缺失、或者离线指标不是线上目标的好 proxy。

Q8：item_id 是否应该进模型？

通常会作为 categorical ID 进 embedding，但要警惕冷启动、长尾、hash 冲突和 ID 记忆过强。大规模系统里会结合 item 语义特征、作者/类目、统计特征来缓解。

Q9：RankMixer 怎么做 ablation？

去掉 token mixing、把 per-token FFN 改共享 FFN、去掉 sequence token、去掉 cross token、dense vs sparse MoE、不同 token 数、不同 model_dim/layers/top-k，分别看 AUC/NDCG/延迟/MFU。

Q10：一句话总结 RankMixer？

RankMixer 是面向工业排序扩容的硬件友好 backbone，它把多域特征 token 化，用 token mixing 做高效跨域交互，用 per-token FFN 保留域内建模，并通过 Sparse-MoE 在参数容量和线上成本之间折中。

9. 面试前 10 分钟速记

必须背下来的主链路

features
-> semantic tokens
-> token mixing
-> per-token FFN / Sparse-MoE
-> pooling
-> CTR/CVR heads
-> business score

必须主动讲出的工业点

  • 不是只追 AUC，要看 P99、吞吐、MFU。
  • 训练和推理 feature schema 必须版本绑定。
  • Sparse-MoE 要看 expert load 和路由稳定。
  • CVR 稀疏要处理 delayed feedback 和校准。
  • RankMixer 和 MMoE/PLE 是 backbone 与 task sharing 的关系。

推荐你的面试表达顺序：先说工业痛点，再讲 token 化和 block 结构，然后对比 MMoE/PLE，最后补离线训练、线上推理、Sparse-MoE 风险。这个顺序比单纯背模型结构更像真实算法工程师。
